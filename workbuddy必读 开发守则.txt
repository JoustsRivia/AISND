# 一、核心架构法则（确保代码可迁移）

1.  **前端统一入口**
    - 所有网络请求（数据查询、文件上传、调用云函数等）必须封装在 `utils/api.js` 中，对外暴露语义化函数（如 `api.login()`、`api.getOrderList()`）。
    - 任何页面或组件**绝对禁止**直接调用 `wx.cloud.callFunction`、`wx.cloud.database`、`wx.cloud.uploadFile` 等平台专属 API。
    - 未来迁移时，只允许修改 `api.js` 的内部实现（改为 `wx.request`），页面代码应能做到零改动。

2.  **云函数内部分层隔离**
    - 云函数内部，将所有依赖 `wx-server-sdk` 的能力封装进独立的 `helpers` 工具文件中：
        - `helpers/user.js`：封装获取当前用户身份的方法（内部调用 `cloud.getWXContext()`）。
        - `helpers/db.js`：封装所有数据库的增删改查操作（内部调用 `cloud.database()`）。
    - 云函数的主逻辑**只能引用**这些 helpers，绝不可直接出现 `cloud.database()` 或 `cloud.getWXContext()` 等原生调用。
    - 核心业务逻辑（校验、计算、状态流转）必须保持纯净，不沾染任何平台特定语法。

3.  **迁移契约**
    - 未来更换服务器时，唯一允许改动的文件是 `api.js` 和云函数内的 `helpers/*.js`。所有页面 UI 代码、云函数中的核心业务逻辑文件，必须能够原样复用。

# 二、分步实施法则（避免上下文爆炸）

当你接到一个整体功能需求时，必须严格按以下流程工作，绝不一次性加载整个项目：

1.  **第一步：理解需求并公布实施计划**
    - 首先将用户的整体需求拆解为若干个相互独立的实施步骤（例如：1. 新建云函数并编写业务逻辑；2. 在 `api.js` 中封装接口；3. 修改 A 页面的 UI；4. 修改 A 页面的交互逻辑）。
    - 在开始任何代码改动前，先清晰地列出这些步骤，让用户确认范围。

2.  **第二步：按步骤顺序执行，每步只加载最少必要文件**
    - 严格按计划顺序执行。在开始某个步骤前，**只读取完成该步骤所必需的文件**。
    - 你必须显式声明将读取哪些文件，并解释为什么这些文件是必要的。确定必要文件的依据如下：
        - **修改页面**：通常需要读取该页面路径下的 4 个文件（.wxml/.wxss/.js/.json），以及 `utils/api.js` 中被该页面引用的相关函数签名。
        - **修改 API 层**：直接读取 `utils/api.js`，以及根据架构法则必须对应的云函数入口文件。
        - **修改云函数**：仅读取该云函数目录下的 `index.js` 及其 `helpers/` 子目录，绝不牵连其他云函数。
        - **涉及全局状态**：仅在需要时读取 `app.js`。
    - 任何情况下，都**禁止**使用 `list files` 或类似操作去获取整个项目的目录树。你只能沿着已知的依赖链（如页面 JS 中的 import 语句）去发现文件。

3.  **第三步：动态增量构建，逐步交付**
    - 完成一个步骤后，你在记忆中仅保留该步骤产出的关键接口摘要（如新增的 API 函数名、云函数名、核心数据结构），而不是完整的文件内容。
    - 进入下一步骤时，如果依赖上一步的产物，可直接基于摘要工作；若确实需要完整代码，再针对性地读取那个文件。
    - 整个过程中，绝不提前加载后续步骤所需的文件。

# 三、融合示例（架构 + 分步）

假设需求是：“实现一个用户注册功能，需填写昵称和手机号，注册成功后跳转到首页。”

你会按以下方式同时贯彻两条法则：

**计划步骤：**
1. 创建云函数 `register`，编写纯净的核心逻辑（通过 helpers 隔离云 API）。
2. 在 `utils/api.js` 中添加 `register` 函数封装。
3. 修改注册页面 `pages/register/register.wxml` 和 `register.js`，调用 `api.register()` 完成交互。

**执行步骤 1：**
- 读取 `cloudfunctions/` 下任一现有云函数的 `helpers/` 结构作为参考（若需要），然后直接创建 `register` 云函数。
- 创建时严格遵循分层：新建 `helpers/user.js` 和 `helpers/db.js`（如果尚未存在则复用公共 helpers，此时可询问用户），主逻辑只调用 helpers，绝不出现 `cloud.getWXContext()` 或 `cloud.database()` 字样。
- 此步骤不读取任何页面或前端 API 文件。

**执行步骤 2：**
- 只读取 `utils/api.js`，在文件末尾追加 `export function register(data) { ... }`，内部调用 `wx.cloud.callFunction({ name: 'register', data })`。
- 不读取任何页面文件。

**执行步骤 3：**
- 只读取 `pages/register/` 下的四个文件，以及 `utils/api.js` 中 `register` 函数的签名。
- 修改 WXML 和 JS：JS 中通过 `import { register } from '../../utils/api'` 调用，UI 绑定事件。
- 不读取其他页面或云函数。

通过以上流程，你才能保证了注册功能 100% 遵循可迁移架构，又确保每一步的上下文极度精简。
